What is a GitHub Codespace? (1-paragraph version)

A Codespace is a Linux computer that runs in GitHub's cloud, with VS Code (a code editor) opened in your web browser. Nothing installs on your laptop — everything happens on the cloud machine. You write code in the browser, save it to GitHub, and it's still there when you close the tab. Free each month — 120 core-hours (about 60 hours of use on the standard 2-core machine) on a personal account, or 180 core-hours (~90 hours) once you verify as a student — then it pauses until the next month. Like Google Docs, but for a programming environment.

Important — when you're done, stop your codespace. Idle codespaces keep using your monthly hours until they auto-suspend (default: 30 min). To stop one immediately: visit github.com/codespaces and click ⋯ → Stop codespace. Leaving a tab open all day will eat through your free tier in under a week.

Storage counts too. Even a stopped codespace keeps using your free storage (15 GB) — GitHub only stops counting when you delete it (⋯ → Delete). A large dataset can hit the storage limit before your hours run out, so delete codespaces you've finished with.

Your first 30 minutes

The shortest path from a blank browser to something real. By the end you'll open one example evidence capsule, change a single number, and watch every estimate, chart and self-audit check update — the fastest way to feel what a self-checking research tool does, and proof the whole thing works.

1. Open your workspace. Click Start in GitHub Codespaces at the top and wait ~2–3 minutes while it builds. A code editor opens in your browser with everything already set up — nothing to install.
2. See it work — open one capsule. Open the living evidence capsules, pick one (say a meta-analysis), change one number in its data, and watch the result, chart and self-audit checks re-run instantly. That's your first success. Each capsule is a single offline page that runs its own statistics and checks itself before asking you to trust it — the quality bar made tangible.
3. Add your free AI key and run a test prompt. In a Codespace the AI agent is already installed — you just add a free key (see Set up your AI key), then ask it: gemini "Summarise the rules in my .claude/rules/ folder". If it answers with the E156, lessons, advanced-stats and rules content, your agent is wired up.
4. Learn the method. The real value is a way of working: brainstorm → spec-lock → plan-lock → TDD → audit. Read STUDENT-WORKFLOW.md for the 5-step path to your first paper, the case studies behind each step, and what UNVERIFIED means (it is not a release pass). Worked-example repos to clone and reproduce: ma-workbench (sglt2i-hfpef), repro-floor-atlas, responder-floor-atlas, impossible-ma.
5. Create your first E156 project. Scaffold a new project with the student new CLI (from e156-student-starter) and aim for one small, reproducible output: one repo + one dashboard + one 156-word micro-paper. Pick a starting point from What you can build below.

What you can build

Once your workspace is open, you can build small but real research artifacts in a few hours. Pick one. Each is sized for a single sitting and produces something you can put on your CV.

1. Mini meta-analysis (E156 paper) Pick 3–5 trials on a topic that interests you (e.g. SGLT2 inhibitors in heart failure, antidepressants in mild depression, ivermectin in COVID). Pool effect sizes with DerSimonian–Laird, write a 156-word E156 micro-paper, ship as a single-file HTML dashboard on GitHub Pages. Asking the agent for the right pooling formula, validating against R metafor, learning to write tight scientific prose.
2. Forest-plot tool from scratch Build a single-file HTML page where the user pastes a CSV of study, effect, lower, upper and gets a forest plot + pooled diamond + I². No backend, pure JS + SVG. SVG, statistical formulas (Q, τ², HKSJ), the discipline of writing tests before code.
3. Risk-of-bias dashboard RoB 2 traffic-light table for a real Cochrane review of your choice. Read the review, score each domain, render a coloured matrix + summary stacked bar. Reading critically, structured data entry, accessible HTML (proper labels, ARIA).
4. CT.gov mini-audit Pull all interventional trials in your country / specialty from the ClinicalTrials.gov v2 API, count completed but no results posted at 2 years. One page, three numbers, one chart. Tip: reading results straight from the open registry when a paper is paywalled is now part of the toolkit's method — see What's new.

Limit you must state in your write-up: CT.gov is one registry. Trials run in Africa are frequently registered with PACTR or ISRCTN instead, and will be invisible to this audit. "Not in this registry" is not "does not exist." A count of "hidden" trials built on a CT.gov-only denominator will over-report hiddenness, and it will over-report it worst for exactly the countries this toolkit is built for. Report your denominator as "CT.gov-registered trials", never as "trials". If you can, cross-check PACTR and ISRCTN by hand and report the overlap you found. 		REST APIs, pagination, dates, honest framing of negative findings.
5. PRISMA flow diagram generator Input: numbers from each screening stage (records identified, duplicates, screened, full-text, included). Output: PRISMA-2020 SVG diagram you can paste into a paper. SVG layout, working backwards from a published standard.
6. Living NMA prototype Pick a 3-arm decision (e.g. ARNI vs ACEi vs ARB in HFrEF). Build a Bucher indirect-comparison engine. Add a "what if a new trial reports HR=X" slider that re-pools live. Network meta-analysis arithmetic (consistency assumption), interactive UI design.
7. Evidence-update bot Cron-style script that re-queries CT.gov for new trials matching a saved PICO and emails / commits a JSON diff. Useful for keeping a review "living". Scheduled jobs, diffing structured data, knowing when to alert vs stay silent.
8. Replication of a published figure Pick a forest plot from a recent NEJM/Lancet meta-analysis. Re-extract the trial table, re-pool, regenerate the figure. Note any discrepancies. Reading methods sections precisely, debugging "why don't I get the same number".

Each project should ship as one GitHub repo + one GitHub Pages dashboard + one E156 micro-paper (156 words). The starter rules teach the agent how to keep all three in sync.

Install on your own machine

Using Codespaces? You can skip this whole section. Everything below is only for installing on your own laptop instead of in the browser. The cloud workspace already has Python, R, and Node set up for you.
Is this safe? Yes — it's open source, and you can read every script before running it. The safest route is GitHub Codespaces: nothing installs on your laptop at all. If you do install locally, the recommended path below is to clone the repo and read the scripts first, rather than pipe a web address straight into your shell. For institutional or shared machines, use the tagged git clone route with the signed release (see Security model) instead of the one-liner.

Step 1 — prerequisites (local install only)

On your own machine, install Python and Node.js first (both free). R is optional — you only need it for the R cross-validation check, and the installer just prints a warning and carries on if R is missing. The installer does stop with a clear message if Python or Node is absent. A 156-word first paper does not need R.

Python 3.11 or newer (required)

Critical: Windows 10/11 ships with a Microsoft Store stub at python.exe that looks like Python is installed but actually opens the Store when you try to use it. You must install real Python from python.org, not the Store. The installer detects the stub and refuses to continue if it's all you have.
1. Download from python.org/downloads (latest 3.x).
2. Run the installer. On the first screen, tick "Add python.exe to PATH" — this is the most-missed step.
3. Click Install Now. It takes about 1 minute.
4. Verify in a new PowerShell window: 
python --version
You should see a Python 3.x version. If you see "Python was not found" or it opens the Microsoft Store, the Store stub is shadowing your install — see Advanced for the one-line fix (turning off the app-execution alias).

Node.js 20 or newer (required)

Needed because Gemini CLI / Claude Code / Codex are all distributed via npm.

1. Download the LTS version from nodejs.org/en/download and run it (default options).
2. Verify in a new PowerShell window: 
node --version
npm --version
You should see v20.x.x or higher.
R 4.5+ (optional — only for the metafor cross-check)

R is used to cross-validate your meta-analysis pools against the gold-standard metafor package. It is entirely optional — skip it and the installer simply prints a warning and moves on; your dashboards still work, they just won't run the extra R cross-check. Add it later if you want that belt-and-braces validation.

1. Download from cran.r-project.org/bin/windows/base/ and run the installer (defaults are fine).
2. Add R to PATH and install metafor — the terminal steps are in Advanced, since they involve editing PATH by hand.

Step 2 — install (clone, read, run)

Recommended — download the installer, read it, then run it. You're about to run a script that writes files into your home folder, so the safest habit (especially if you've never used a terminal) is to get the source first, look at it, and run it from disk — not pipe a web address straight into your shell.

git clone https://github.com/mahmood726-cyber/e156-ecosystem-starter
cd e156-ecosystem-starter
# open and skim install/install.ps1 (Windows) or install/install.sh (macOS/Linux/WSL)
# — both are short and commented — then run the one for your system:
.\install\install.ps1        # Windows PowerShell
bash install/install.sh      # macOS / Linux / WSL

The installer asks a few Yes/No questions (all default to Yes) and finishes in about 2 minutes. Confirm it worked by listing the installed rules: ls ~/.claude/rules (macOS/Linux) or ls $env:USERPROFILE\.claude\rules (Windows) — you should see four .md files (e156, lessons, advanced-stats, rules).

Prefer a one-click download instead? Grab the installer file and run it:

Download .bat (Windows) Download .sh (macOS / Linux)

On Windows the .bat may be blocked by Smart App Control — see Advanced for the fix. On macOS/Linux: chmod +x e156-setup.sh && ./e156-setup.sh.

One-liner install (faster, but pipes a remote script into your shell)

If you understand the trade-off — you're running code straight from the web before reading it — the one-liners are:

irm https://mahmood726-cyber.github.io/e156-ecosystem-starter/bootstrap.ps1 | iex   # Windows PowerShell
curl -fsSL https://mahmood726-cyber.github.io/e156-ecosystem-starter/e156-setup.sh | bash   # macOS / Linux / WSL

The clone-and-read path above is preferred, especially for a first-time terminal user. The source is open and auditable: bootstrap.ps1, install.sh.

Set up your AI key (agent CLI)

The starter rules and memory are designed for three agents. You only need one to get going. If you're starting from zero, install Gemini CLI — the free tier is the most generous.

In a Codespace, Gemini CLI is already installed for you. You can skip straight to step 3 below — getting your free API key — and set it with export GEMINI_API_KEY="your-key" in the terminal.

Option A: Gemini CLI (recommended — free tier)

1. Install Node.js 20 or newer (Windows installer, default options).
2. Open a fresh PowerShell window and install the CLI globally: 
npm install -g @google/gemini-cli
3. Get a free Google API key (no credit card needed):
  1. Go to aistudio.google.com/app/apikey and sign in with any Google account.
  2. Click Create API key → Create API key in new project.
  3. Copy the long key (starts with AIza...). Treat it like a password — never paste it into a public repo or chat.
4. Tell Windows the key is yours so the CLI can find it: 
setx GEMINI_API_KEY "paste-your-key-here"
Close and reopen your terminal so the new variable loads.
5. Test it: 
gemini "summarise this file" < README.md
If you get a coherent answer, you're done.
Free-tier limits (Google AI Studio): the free tier needs no credit card, and Gemini Flash is usually enough for small research tasks. The exact limits change by model and by account, so check your active limits in Google AI Studio (or the rate-limits page). If you hit a wall, the CLI tells you when the quota resets.

Option B: Claude Code

Paid (Anthropic API or Claude Pro/Max). Best at long-context work. Install:

npm install -g @anthropic-ai/claude-code
claude login

See docs.anthropic.com/en/docs/claude-code.

Option C: OpenAI Codex

Free with a ChatGPT Plus account (uses your subscription quota), or pay-per-token via API key. Install:

npm install -g @openai/codex
codex

On first run it walks you through sign-in. See github.com/openai/codex.

What this toolkit cannot do

Every tool here states its blind spot next to its feature, because a tool that hides its blind spot is the thing this project exists to replace. Read this section before you trust a number it gives you.

  • One-registry checks over-report hiddenness. Any check built on ClinicalTrials.gov alone cannot see PACTR, ISRCTN, CTRI, or ChiCTR registrations. It will falsely flag African- and Asian-registered trials as missing. See Project 4 above.
  • Cross-family agreement is weaker than it looks. A panel of models drawn from different families gives roughly two effective independent votes, not one per model. Do not read panel agreement as replication.
  • Our nulls, published: seven novelty claims we tested did not survive; six forensic bids returned clean nulls. These are findings, not embarrassments — they are here so you do not spend your time re-running them.
  • Coverage floors are usually our failure, not the world's. When this toolkit reports "no data available", treat that as a statement about the toolkit's reach until you have checked a second source yourself.

Prior art — we are mostly not first

Assume this project is last until shown otherwise. Named, so you can go read the originals and judge for yourself:

  • Weak supervision / labelling rules — Snorkel and the data-programming literature. Our Tier-1 rules are this idea, applied to evidence synthesis.
  • Model cascades for cost — FrugalGPT. Our cheap-check-first routing is this idea.
  • Document-processing harnesses — ScriptoriumWS / Alfred.
  • Estimating what you missed — capture–recapture methods, long-standing in software engineering and ecology before we applied them here.
  • Registry-vs-publication discrepancy — Hartung & Zarin (2014); Wieseler (2013) on regulatory data completeness. They found it first.
  • Missing-data handling in synthesis — Mavridis (2020).

What is ours: the assembly — offline-first, open-data-only, fail-closed, runnable on a weak laptop with no API key and no paywalled source. Not the parts.

The plain-language intro at the top calls this "AI rules, quality checks, and project memory." Here are the real component names, what each does, and the security model — for supervisors, IT staff, and the curious.

What's new in the toolkit

The starter keeps moving. If you installed an earlier version, here's what has been added since — all of it still offline-first and gentle on a free-tier quota, because that's the constraint most students are working under.

Spend fewer tokens — reuse what the kit already ships

Three small offline helpers (no internet, no AI, no cost) cut the biggest hidden drain on your free requests: asking an agent to re-write something the ecosystem already contains. reuse.py finds an existing function to copy, recall.py pulls only the saved notes relevant to your question instead of loading your whole memory file, and find-related-repos.py checks whether you've built something similar before. See the token-budget guide for where your free requests actually go.

Turn a trial into an analysis config — now across 17 fields

The RCT extractor now auto-detects 17 disease areas, including many that matter most where this toolkit is used — HIV, malaria, typhoid, schistosomiasis, sickle cell, cholera, TB, maternal/neonatal care, and more. Paste an abstract, get a starter meta-analysis config. The student path is standard-library-only (no heavy installs); the large PDF/science dependencies stay optional. Source: rct-extractor-v2.

Rebuild the whole meta-analysis toolkit — offline

Five optional layers now let you recreate the full workflow end-to-end without spending tokens, each with a zero-cost path you can just open and read:

  • RapidMeta kit — turns a small JSON config into a self-contained meta-analysis dashboard (standard-library only, offline).
  • AACT cockpit & AACT kit — analyse ClinicalTrials.gov / AACT data on your own machine, or just read the worked examples.
  • Pairwise70 workbench — a static gallery that reproduces each analysis in your browser, no Python needed.
  • E156 capsule + chart-kit — the 7-sentence paper contract plus a ~120-line pure-SVG forest-plot kit and a pre-baked sample. The lowest-footprint layer in the ecosystem.

These sit alongside the 40 living evidence capsules already on this page.

A verifier that checks your meta-analysis math

The Overmind verifier is now at a stable v1.0.0 release (pinned to a signed tag) and has gained an evidence-synthesis subsystem: a deterministic pooling engine and a gold-benchmark that reproduces a published pooled estimate and fails closed if the numbers don't match, plus screening-recall calibration, claim-grounding, and PRISMA-flow checks. Same idea as the capsules — don't trust a number until it has checked itself. Source: overmind.

Method and standards you can map your own work to

The rules that guide the agent picked up new method guidance too — for example, requiring a held-out test split before any “we tuned or optimised it” claim, and a just-in-time rule index so the agent loads only the rules it needs for the task in front of it. The method also encourages an open-data-only, provenance-verified way of correcting published results: rebuild a number from freely-available sources (PubMed, ClinicalTrials.gov, Europe PMC), and where it doesn't hold up, add a corrected value beside the original with a link back to its source — never overwrite. These checks are written to follow and cite recognised standards — the Cochrane Handbook, Chapter 6 (the right effect measure for the kind of data), GRADE indirectness, and the transitivity assumption — and the whole ecosystem is mapped to the 2025 RAISE framework for responsible AI in evidence synthesis, so you can see — and learn — where your own project meets the bar.

Staying current

To pull the latest starter without losing your memory or your own edits, run update-ecosystem.ps1 (add -Full to also refresh Sentinel, Overmind, and ProjectIndex). Think of it as the update button for the whole kit.

Honest about what's new. The individual methods here — pooling engines, GRADE, provenance tracking, integrity checks — all exist in the literature, and this builds on that prior art. What the starter adds is putting them together into one offline-first, open-data-only, self-checking toolkit a student can actually run on a free plan. Same rule as everywhere else on this page: real, named, runnable pieces behind each claim — no overclaiming.

Advanced / power users

Terminal-heavy bits a first-time user can safely ignore. Grouped here so the main path stays clean.

Python opens the Microsoft Store instead of running

The Store stub is shadowing your real install. Turn it off: Windows Settings → Apps → Advanced app settings → App execution aliases → toggle off python.exe and python3.exe, then reopen PowerShell.

Finish the optional R setup (PATH + metafor)

The R installer doesn't add R to PATH. In PowerShell, point PATH at the newest R's bin:

$rDir = (Get-ChildItem 'C:\Program Files\R' -Directory | Sort-Object Name -Descending | Select-Object -First 1).FullName + '\bin'
[Environment]::SetEnvironmentVariable('Path', "$env:Path;$rDir", 'User')

Close and reopen PowerShell, confirm Rscript --version, then install the validation package (once per machine):

Rscript -e "install.packages('metafor', repos='https://cloud.r-project.org')"

Windows Smart App Control blocks the .bat

Windows 11 Smart App Control (different from SmartScreen) blocks unsigned downloaded scripts without a "Run anyway" button. In order of effort:

  1. Use the clone-and-read path in Install on your own machine — Smart App Control trusts git.exe, so cloning and running .\install\install.ps1 is not blocked.
  2. Unblock the .bat manually: right-click e156-setup.bat → Properties → tick Unblock → OK, then double-click again. Or in PowerShell: Unblock-File ~\Downloads\e156-setup.bat and run it.
  3. Use the PowerShell one-liner (in the Local install one-liner box) — it runs in memory without writing an unsigned file to disk.

Full install / CI / unattended

.\install\install.ps1 -Full            # all optional layers (Sentinel, Overmind, ProjectIndex...)
.\install\install.ps1 -NonInteractive  # no prompts, for CI / scripted setup

What if something goes wrong

  • Python not found → install from python.org, tick Add Python to PATH. If it opens the Store instead, see Advanced.
  • Download blocked → check your firewall isn't blocking github.com.
  • Install failed mid-way → the installer writes a transcript to %LOCALAPPDATA%\e156\logs\install-<ts>.log. Open an issue with that log attached, or run .\scripts\doctor-report.ps1 to package it.
  • SmartScreen or Smart App Control blocked the .bat → use the clone-and-read install in Install on your own machine, or see Advanced → Smart App Control. SmartScreen's More info → Run anyway works for it; Smart App Control has no override, so the clone path or the in-memory one-liner is the way through.

Security model — what the SHA gate does and doesn't protect

The SHA gate is for download integrity, not source integrity. Specifically:

  • What it catches: tampering of install.ps1 / install.sh between the GitHub Pages CDN and your laptop (corrupted download, MitM injection on a hostile network). The installer hashes itself against docs/HASH.txt / docs/HASH-linux.txt fetched from the same origin and refuses to run on mismatch.
  • The bootstrap entrypoints (bootstrap.ps1, e156-setup.bat, e156-setup.sh) are not SHA-gated. They run before the gate fires; they then fetch the rest of the repo and the SHA-gated install.ps1 validates itself. So a tampered bootstrap script could short-circuit the gate entirely. The bootstrap layer is small and auditable — read the source via the links below before running it on a security-sensitive machine.
  • What it does not catch: compromise of the GitHub repo itself. A bad actor with push access to mahmood726-cyber/e156-ecosystem-starter can update install.ps1 AND docs/HASH.txt together and the gate will pass. Any curl|bash-style installer has this property.
  • For high-trust installs (e.g. a department lab with sensitive data), prefer the git-clone path with a tagged release: git clone --branch v0.x.y --depth 1, verify the tag's GPG signature, then run .\install\install.ps1 from the clone. The signed tag is a trust anchor independent of the Pages CDN.
  • What runs locally: the install touches ~/.claude, ~/.gemini, ~/.codex, and (if you opt in) %USERPROFILE%\code\. It does not require admin rights, does not modify system PATH, and rolls back on failure. The full source is auditable on GitHub.
  • Sub-installer pinning: the optional Sentinel and Overmind chains pip-install from GitHub. Both are pinned by default (SENTINEL_REF=v0.1.0, OVERMIND_REF=<known-good-SHA>) so a fresh install is reproducible even if upstream main drifts. To opt into bleeding-edge or roll back a release: set $env:SENTINEL_REF / $env:OVERMIND_REF to any branch, tag, or commit SHA before running the chain installer.
